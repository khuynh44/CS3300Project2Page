Introduction/background

The purpose of this project is to allow users to play 24. Traditionally played with playing cards, 24 is a game where the player(s) are given four natural numbers, and must use ordinary math operations (addition, subtraction, multiplication, division) in order to make the number 24. The user will be given four numbers, and will attempt to make 24. When they are finished, then they will be given another set of four numbers, and they must, once again, make 24. They will do this as many times as they can within a given time frame. The user will then get a score based on how many they got correct; their high score will be stored with their account, and they will be able to see a leaderboard with each users high score ranked.

Motivation, Project Scope, Project Management

Due to time constraints, our team decided to focus on two game modes that we believe to be the most enjoyable for the user. Future work that we would like to do would involve developing new game modes or even embedding this game into a larger collection of similar obscure math/puzzle games.

We utilized GroupMe for team communications and facilitated team meetings via Zoom video conferences. We initially intended on using a ZenHub board for project management, but this proved to be unnecessary and was instead replaced with a simple living word document with week-to-week goals.

Software Technologies

Development of the frontend product will be completed in VS code, using vanilla Javascript, HTML, and CSS as well as an open-source code algorithm. Backend development will use Java and be completed in IntelliJ. The project will be hosted using Google App Engine, and user data will be stored using Google Cloud SQL. Version control for our project will be done in GitHub, integrating smoothly with our choice of IDE’s. Project management and task tracking will utilize the ZenHub extension, and comprehensive testing for the project and its functions will be written alongside the code using JUnit tests.

Project Lifecycle

Our team used an Agile software development strategy that was divided into 4 sprints. The sprint goals were Ideation & Requirements Analysis; Repository Creation, Algorithm Development, & Initial Implementation; Frontend & Backend Implementation; and Alpha Testing, Beta Testing, & Unit Testing, respectively. Each sprint was one week long and concluded with a team meeting to discuss any challenges, new directions, requirements changes, and the work that we wanted to focus on in the following sprint. The meetings were led by our project manager, Chris, and each teammate was expected to give a progress report and discuss any blockers. We maintained two different GitHub repositories for the project, one for the frontend and one for the backend. These two components were deployed as independent Google Cloud applications connected to each other via API calls.

Requirements (Functional and Non-Functional)

Our primary functional requirement was for our users to be able to play the game 24. This encompasses several functional requirements, including a user's ability to log in to the web app, interact with the game and submit an answer, recieve a response as to whether they were correct, and view their previous highscore. From a system perspective, this also includes the product's ability to log a user in and out and store and retrieve each user's highscore individually.

Our product's non functional requirements include the system's ability to change pages quickly on the average network, responsiveness to inputs and immediate feedback, and easy maintenance through a simplified codebase. Due to the minimal number of communications between frontend and backend, the site can support a significant number of users simultaneously, resulting in good scalability. Additionally, user's passwords are encrypted such that if an attacker gains access to our database, they will not be able to pull user's passwords.

Testing - Test Strategy, test cases and outcomes

Our testing consisted of a blackbox portion that involved alpha and beta testing as well as a whitebox portion that involved unit tests. Alpha testing was performed by each team member and beta testing was performed by a friend of each team member after we fixed bugs that were found during the alpha testing. There was a checklist that we made to evaluate the system:

  • Create user
  • Login as user
  • Read “How to Play”
  • Play timed run
  • Play blitz round
  • Check leaderboard

Beta testing revealed that the users believed that the application met the minimum requirements of each list item, but that the UX was not ideal for a game that requires the user to play against a time limit. Future work will include improving the user interface to make it easier to use quickly.

The whitebox testing consisted of 15 unit tests that were performed using a Python script. The test cases provide ~87% statement coverage and are outlined below:

Test Case What functionality is being tested? Outcome: (P)ass/(F)ail
1. InitialPage The initial page loads, indicating that the server is online and functioning P
2. CreateUser The user can create a new account. P
3. CreateUserNameExists The user cannot create a new account with an existing username. P
4. ShortPassword The user cannot create a new account with a password that is too short (<5 characters) P
5. LoginValid The user’s valid credentials are verified successfully. P
6. LoginInvalidPassword Users are denied access when they try to login in with the right username but wrong password. P
7. LoginInvalidUsername The system will invalidate a login with a username that has no associated account. P
8. UpdateUserInfo (part 1) New user’s data will be updated correctly (i.e. the fastest time should no longer be null and number of games played/won are incremented) P
9. UpdateUserInfo (part 2) The user’s data from part 1 should be updated and the fastest time should be changed. P
10. UpdateUserInfo (part 3) The user’s data from part 2 should be updated and the fastest time should not be changed. P
11. UserInfoNoGamesPlayed If a game was not played, then the user’s data should not be changed. P
12. GetLeaderBoard Successfully displays the top scores among users sorted in the order of fastest time ascending. P
13. DeleteUsers All users are successfully deleted from the database. P
14. DeleteAllWrongPassword If the wrong password is passed to the /delete endpoint, then nothing is deleted. P
15. GetAllUsers All users’ data is returned from the backend. P
